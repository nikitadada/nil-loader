# nil-loader

**gRPC Load Testing Tool** с real-time дашбордом и автоматическим определением точки деградации.

### Слайд 2 — Зачем нужно нагрузочное тестирование

**Проблема:** Сервис работает на 10 пользователях, но что будет на 10 000?

Нагрузочное тестирование отвечает на вопросы:
- Какой **максимальный RPS** выдерживает сервис?
- При какой нагрузке начинается **деградация**?
- Какой RPS **безопасен** для продакшена?
- Где **узкое горло**: CPU, память, сеть, база?

Существующие инструменты (ghz, k6, Locust) не всегда поддерживают gRPC «из коробки», требуют кодогенерации или не дают real-time визуализацию.

> Картинка: схема «клиент → load tester → gRPC сервис», стрелки с RPS.

---

### Слайд 3 — Базовые термины

| Термин | Описание |
|---|---|
| **RPS** (Requests Per Second) | Количество запросов в секунду — основная метрика нагрузки |
| **Latency** | Время ответа на один запрос (в миллисекундах) |
| **p50 / p95 / p99** | Перцентили латентности. p99 = 99% запросов быстрее этого значения |
| **Throughput** | Реальная пропускная способность (фактический RPS) |
| **Concurrency** | Количество параллельных соединений / горутин |
| **Error Rate** | Процент неуспешных запросов |
| **Точка деградации** | RPS, при котором латентность или ошибки начинают расти нелинейно |
| **HDR Histogram** | Алгоритм для точного расчёта перцентилей без большого потребления памяти |

> Картинка: график с тремя линиями p50/p95/p99, показывающий «расслоение» при нагрузке.

### Слайд 5 — Фишка 1: Dynamic gRPC без кодогенерации

**Проблема:** Обычные gRPC-клиенты требуют `protoc` + генерацию Go-кода под каждый сервис.

**Решение в nil-loader:**
- **gRPC Reflection** — указываем адрес сервиса → автоматически получаем список сервисов и методов
- **Загрузка .proto на лету** — вставляем proto-определение прямо в UI
- **Dynamic Message** — формирование запросов без генерации кода (`jhump/protoreflect`)

Пользователь просто вводит `localhost:50051`, нажимает **Discover** и видит все доступные методы.

> Скриншот: боковая панель с результатом Discover via Reflection — список сервисов и методов.

---

### Слайд 6 — Фишка 2: Динамическое управление нагрузкой

**Два режима:**

| Режим | Описание |
|---|---|
| **Constant** | Фиксированный RPS на всё время теста |
| **Ramping** | Плавный рост от Start RPS до End RPS за указанное время |

**Worker Pool** с ограничением concurrency — не «кладём» машину-тестировщик.

Ramping позволяет найти точку отказа: нагрузка растёт линейно, и мы видим момент, когда сервис перестаёт справляться.

> Скриншот: график Throughput (RPS) — Target vs Actual, где видно расхождение при высокой нагрузке.

---

### Слайд 7 — Фишка 3: Генератор данных

**Проблема:** Тестирование одинаковым payload → кэширование → нереалистичные результаты.

**Решение — шаблоны с Faker-токенами:**

```json
{
  "user_id": "{{faker.uuid}}",
  "name": "{{faker.name}}",
  "email": "{{faker.email}}",
  "phone": "{{faker.phone}}"
}
```

Каждый запрос получает **уникальные данные**.

Доступные токены: `uuid`, `email`, `name`, `firstName`, `lastName`, `phone`, `username`, `sentence`, `timestamp`, `int`, `bool`.

**CSV-источник:** загрузка данных из CSV для параметризации запросов (`{{csv.column}}`).

> Скриншот: поле Payload Template в UI с faker-шаблоном.

### Слайд 8 — Фишка 4: Real-time дашборд

Четыре виджета обновляются каждую секунду через WebSocket:

| Виджет | Что показывает |
|---|---|
| **Latency Chart** | Три линии: p50, p95, p99. Видно «расслоение» при деградации |
| **Throughput** | Target RPS vs Actual RPS. Расхождение = сервис «захлебнулся» |
| **Success Rate** | Donut: OK vs ошибки + данные о деградации |
| **Error Breakdown** | Группировка по gRPC-кодам (Unavailable, DeadlineExceeded, Internal) |

Плюс:
- **Error Log** — последние ошибки с кодом, сообщением и таймстемпом
- **Streaming Logs** — терминал (xterm.js) с логами движка в реальном времени

> Скриншот: полный UI во время активного теста — все графики заполнены данными.

---

### Слайд 9 — Фишка 5: Автоматическое определение точки деградации

**Как работает детектор:**

1. Первые 5 секунд — сбор **baseline p99** (нормальная латентность)
2. Каждую секунду проверяем:
   - p99 вырос > **2.5x** от baseline?
   - Интервальный error rate > **15%**?
3. Если да — фиксируем **точку деградации**

**Результат:**

| Метрика | Описание |
|---|---|
| **Degradation at** | RPS, при котором началась деградация |
| **Recommended** | Безопасный RPS = 80% от точки деградации |
| **Max stable** | Максимальный RPS при стабильной работе |

> Скриншот: блок Success Rate & Degradation с красным бейджем «DEGRADATION DETECTED» и тремя метриками.

### Слайд 10 — Телеметрия под капотом

**HDR Histogram** (`hdrhistogram-go`):
- Точные перцентили (p50/p95/p99) без хранения каждого значения
- Диапазон 1µs – 30s, 3 значащие цифры
- Интервальная гистограмма сбрасывается каждую секунду

**Error Classifier:**
- Автоматическая группировка по gRPC status codes
- `Unavailable`, `DeadlineExceeded`, `Internal`, `Unknown` и др.
- Кумулятивные и интервальные счётчики

**Snapshot (1 сек):**
```
timestamp | p50 | p95 | p99 | targetRps | actualRps | success | errors | errorBreakdown
```

> Картинка: нет (данные в самом слайде).

### Слайд 11 — Демо: тестируем Example Service

**Тестовый gRPC-сервис** (встроен в проект):
- Метод `ProcessUser` — принимает user_id, name, email, phone
- До 800 RPS: ~5% ошибок, латентность 5–50ms
- Выше 800 RPS: ошибки растут экспоненциально, латентность — квадратично
- Имитирует реальную деградацию

**Сценарий демо:**
1. Discover via Reflection → выбрать ProcessUser
2. Вставить faker-шаблон
3. Ramping 100 → 1500 RPS, 60 секунд
4. Наблюдать деградацию в реальном времени
5. Посмотреть точку деградации ~800–900 RPS

> Скриншот: UI после завершения Ramping-теста — графики с видимой деградацией, бейдж с результатами.

### Слайд 15 — Итоги

**nil-loader** решает задачу нагрузочного тестирования gRPC-сервисов:

- **Без кодогенерации** — reflection + dynamic messages
- **С умной нагрузкой** — constant / ramping + worker pool
- **С уникальными данными** — faker + CSV шаблоны
- **С точной телеметрией** — HDR Histogram, посекундные snapshot'ы
- **С real-time UI** — графики, логи, ошибки через WebSocket
- **С автодетекцией деградации** — точка отказа, рекомендуемый RPS

Один бинарник. Ноль конфигурации. Открыл браузер — тестируешь.

> Скриншот: финальный вид UI с завершённым тестом и определённой точкой деградации.

```bash

```
